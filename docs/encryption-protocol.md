# Encryption Protocol — AES-GCM End-to-End Encryption

## Overview

All file data transferred over the WebRTC DataChannel is encrypted end-to-end using AES-GCM. The encryption key is generated by the sender and transmitted to the receiver via the URL hash fragment. The signaling server never sees the encryption key.

---

## Key Generation

### Sender (Key Creator)

```typescript
// Generate a 256-bit AES-GCM key
const key = await crypto.subtle.generateKey(
  { name: 'AES-GCM', length: 256 },
  true,   // extractable
  ['encrypt', 'decrypt']
);

// Export as raw bytes for URL encoding
const rawKey = await crypto.subtle.exportKey('raw', key);

// Encode as base64url (no padding) for URL hash
const keyParam = base64urlEncode(new Uint8Array(rawKey));
// Result: "https://example.com/room/ABC123#key=<keyParam>"
```

### Receiver (Key Recipient)

```typescript
// Extract key from URL hash
const keyParam = window.location.hash.replace('#key=', '');

// Decode and import
const rawKeyBytes = base64urlDecode(keyParam);
const key = await crypto.subtle.importKey(
  'raw',
  rawKeyBytes,
  { name: 'AES-GCM', length: 256 },
  false,  // not extractable
  ['decrypt']
);
```

## Encryption Scheme

### Per-Chunk Encryption

Every chunk is encrypted independently. This enables:
- Parallel encryption/decryption
- Random access for resume
- No ciphertext expansion accumulation

```typescript
async function encryptChunk(chunk: Uint8Array, key: CryptoKey, sequenceNumber: number): Promise<Uint8Array> {
  // Derive IV from sequence number (12 bytes)
  const iv = new Uint8Array(12);
  iv.set([
    (sequenceNumber >> 24) & 0xFF,
    (sequenceNumber >> 16) & 0xFF,
    (sequenceNumber >> 8) & 0xFF,
    sequenceNumber & 0xFF,
    0, 0, 0, 0,  // padding
  ], 0);
  // Bytes 4-11 are zeroed to avoid IV collision across different files
  // (sequence resets per file)

  // Encrypt
  const encrypted = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv, additionalData: undefined, tagLength: 128 },
    key,
    chunk
  );

  return new Uint8Array(encrypted);  // ciphertext + 16-byte GCM tag
}
```

### IV Structure

```
Byte 0-3:     Sequence number (big-endian uint32)
Byte 4-11:    Zero padding (reserved)
Total:        12 bytes (96 bits, standard AES-GCM IV)
```

**Rationale**: Using the sequence number as IV ensures uniqueness per chunk within a transfer. The 8 bytes of zero padding prevent IV collisions across chunks while maintaining a deterministic, communication-free IV agreement.

### Decryption

```typescript
async function decryptChunk(encrypted: Uint8Array, key: CryptoKey, sequenceNumber: number): Promise<Uint8Array> {
  const iv = new Uint8Array(12);
  iv.set([
    (sequenceNumber >> 24) & 0xFF,
    (sequenceNumber >> 16) & 0xFF,
    (sequenceNumber >> 8) & 0xFF,
    sequenceNumber & 0xFF,
    0, 0, 0, 0,
  ], 0);

  const decrypted = await crypto.subtle.decrypt(
    { name: 'AES-GCM', iv, tagLength: 128 },
    key,
    encrypted
  );

  return new Uint8Array(decrypted);
}
```

## Key Transmission via URL Hash

```
Format: https://<host>/room/<roomId>#key=<base64url-encoded-key>

Example: https://p2p-share.vercel.app/room/ABC123#key=Aa1B2Cc3Dd4Ee5Ff6Gg7Hh8Ii9Jj0Kk1Ll2Mm3Nn4Oo5Pp
```

**Properties**:
- Hash fragment (`#...`) is never sent to the server in HTTP requests
- The frontend reads it via `window.location.hash`
- Server logs never contain the key
- Referrer headers do not include the hash
- Lost if user copy-pastes URL without hash — receiver will see a "missing key" error

## Encryption Overhead

Per chunk:

```
Original chunk:     N bytes
Encrypted output:   N + 16 bytes (GCM authentication tag)
Encryption IV:      0 bytes (deterministic, not transmitted)
Total overhead:     16 bytes per chunk
```

For a file with 16 KB (16384 byte) chunks:
- Overhead: 16 / 16384 = 0.098% per chunk
- Total overhead for 100 MB file: ~100 KB

## SHA-256 Verification

In addition to per-chunk encryption, the entire file is hashed with SHA-256:

```typescript
async function hashFile(file: File): Promise<string> {
  const buffer = await file.arrayBuffer();
  const hashBuffer = await crypto.subtle.digest('SHA-256', buffer);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
}
```

**Integrity flow**:
1. Sender computes SHA-256 of original file (before encryption)
2. Sender sends SHA-256 hash via FILE_META (in the clear)
3. Receiver computes SHA-256 of reassembled original file (after decryption)
4. Receiver compares hashes — if mismatch, transfer is flagged as failed

## Threat Model

| Threat | Mitigation |
|--------|------------|
| Signaling server compromise | Key never sent to server (URL hash) |
| MITM on WebRTC | DTLS-SRTP + per-chunk GCM authentication |
| Hash collision | SHA-256 (2^128 collision resistance) |
| Chunk replay | Sequence-based IV prevents replay |
| Chunk reordering | Sequence numbers in DataChannel + IV binding |

---

## Limitations

- Encryption key in URL hash is visible in browser history and bookmarks (if saved)
- No forward secrecy — if key is compromised, all past transfers are readable
- URL hash is not encrypted in transit (but HTTPS protects the full URL)
- Large file hashing loads the main thread (mitigation: Web Workers for MVP)

---

## Future Improvements

- Diffie-Hellman key exchange for forward secrecy
- Per-file ephemeral keys
- Web Worker offloading for hash computation
- Chunk-level MAC in addition to file-level SHA-256
